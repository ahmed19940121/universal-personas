# Contributing

Thanks for considering a contribution. This framework benefits from input, and the more contexts it travels into the more useful it gets. A few conventions to keep it coherent.

## What this framework is (and isn't)

**It is:** a portable, motivation-centric adaptation of the dev-personas framework, designed to travel across industries and product contexts.

**It isn't:** a job-title directory, a demographic segmentation, or a replacement for primary research on your actual users. Use it as scaffolding; validate against real humans.

## Contribution types that are welcome

- **Sub-personas** for the four archetypes that don't yet have them: Architect, Explorer, Connector, Steady
- **Worked examples** applying the framework to a specific product, industry, or channel
- **Corrections and refinements** to existing persona documents
- **Translations** of any document into other languages
- **Visual assets** — diagrams, icons, generated illustrations
- **Real-world validation** — case studies of the framework being used, what worked, what didn't

## Contribution types to think twice about

- New parent personas. Five is deliberate; the axis model doesn't really support a sixth without reducing clarity.
- Tweaking the axes. Possible, but requires consensus across all personas simultaneously.
- Persona renames. Names are load-bearing; they're the shorthand the framework is used with.

## Conventions

### Naming
- Parent personas use an evocative noun: *The Architect, The Explorer, The Builder, The Connector, The Steady*.
- Each persona has a single first name starting with their archetype letter: **A**dam, **E**lan, **B**ilal, **C**ora, **S**oren.
- Sub-personas follow the same first-letter rule (Bayo, Brynn, Benedict for Builder; propose one for Architect starting with A, and so on).

### Document structure
Match the section order in the existing persona files:

1. Quote + Core Identity
2. Key Motivations
3. Profile (table + where-to-find)
4. Technology Evaluation Criteria
5. Engagement Approach (strategies, frustrations, red flags)
6. Team-Specific Engagement Strategies (Content & Education, Marketing & Messaging, Product & Sales)
7. Content examples
8. Contextual Variations (life stage, role context, domain nuances)
9. Licensing footer

### Tone
- Specific over generic. "Cora works on her phone between meetings" beats "Cora is a busy professional."
- Honest about limitations. Each persona has red flags and frustrations for a reason.
- No corporate-speak. The personas describe humans. Write about them as humans.
- British English spelling (*-ise, colour, behaviour*) for consistency with this fork.

### Axis positions
Don't move an existing persona's axis position in a PR. If you think a position is wrong, open an issue first — the position has downstream implications for every section of the document.

## Pull request checklist

- [ ] New or changed persona files follow the section order above
- [ ] Names follow the first-letter convention
- [ ] Any new sub-persona is explicitly linked from its parent persona file
- [ ] License footer is present and accurate
- [ ] British English spelling throughout
- [ ] No claims about real named individuals unless they've consented

## Issues

Issues are welcome for:
- Proposing a sub-persona set for one of the archetypes that doesn't yet have one
- Flagging inconsistencies or contradictions across documents
- Suggesting worked-example applications
- Reporting diagram or rendering problems

Please keep issues focused on one thing at a time — it makes them easier to resolve.

## Credit

Every merged contribution will be acknowledged in the contributors list. If you'd prefer to be credited pseudonymously or not at all, mention it in your PR.
